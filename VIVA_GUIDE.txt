1️⃣ DYNAMIC PROGRAMMING: 0/1 Knapsack Selection
FILE: backend/algorithms/dp_shortlist.py

🔧 ALGORITHM:
  Selects candidates to maximize total fusion score within a hiring budget.
  Each candidate has: name, fusion_score, hiring_cost (salary band).
  DP table: dp[i][w] = max score using first i candidates with budget w.
  
  RECURRENCE RELATION:
    dp[i][w] = max(dp[i-1][w],                    // Don't take candidate i
                   dp[i-1][w-cost[i]] + score[i]) // Take candidate i

  APPROACH: Bottom-up tabulation (no recursion overhead).
  Backtrack to reconstruct which candidates were selected.

💡 VIVA ANSWER (2 sentences):
  "We used 0/1 Knapsack DP because the company has a fixed hiring budget and
   wants to select the best candidates (by fusion score) without exceeding it—
   a classic optimization problem where each candidate is either hired or not."

⏱️  COMPLEXITY:
  Time:  O(n × budget) where n = candidates
  Space: O(n × budget) for DP table

📊 DEMO OUTPUT:
  Budget: 6 → Selected: ['Diana', 'Charlie', 'Bob'] → Total Score: 253

2️⃣ GRAPH + BFS: Skill Gap Finder
FILE: backend/algorithms/skill_graph.py

🔧 ALGORITHM:
  Builds a directed skill prerequisite graph (ML → Python, ML → Statistics).
  Given current skills and target job requirements, uses BFS to find:
    1. Missing skills
    2. Shortest learning path (fewest prerequisites to master)
  
  Graph structure: skill → [prerequisites]
  Example: ML = ["Python", "Statistics"] means learn Python & Statistics first.

💡 VIVA ANSWER (2 sentences):
  "We used BFS on a skill prerequisite graph because it efficiently finds the
   shortest learning path from current skills to target job requirements—BFS
   guarantees the minimum number of intermediate skills to learn."

⏱️  COMPLEXITY:
  Time:  O(V + E) where V = skills, E = prerequisite edges
  Space: O(V + E) for graph + queue

📊 DEMO OUTPUT:
  Current: ['Python', 'SQL'] → Target: ['ML', 'Kubernetes']
  Missing: ['Kubernetes', 'ML']
  Learning Path: ['Python', 'ML']

3️⃣ GREEDY: Activity Selection Interview Scheduling
FILE: backend/algorithms/interview_scheduler.py

🔧 ALGORITHM:
  Classic greedy activity selection: schedule max non-overlapping interviews.
  Approach:
    1. Sort candidates by end_time (earliest finish first)
    2. Iterate through sorted list
    3. Greedily pick candidate if start_time >= last_end (no overlap)

  Why greedy works: By picking the candidate finishing earliest, we leave
  maximum room for future interviews—proven optimal by exchange argument.

💡 VIVA ANSWER (2 sentences):
  "We used greedy activity selection because it maximizes the number of
   non-overlapping interviews in a fixed schedule—sorting by end time and
   greedily selecting earliest-finishing candidates always gives the optimal count."

⏱️  COMPLEXITY:
  Time:  O(n log n) for sorting, O(n) for selection
  Space: O(n)

📊 DEMO OUTPUT:
  Input: 6 candidates with overlapping/non-overlapping slots
  Scheduled: 4 interviews (Alice, Bob, Diana, Frank)

4️⃣ DIVIDE & CONQUER: Merge Sort with Rank Delta
FILE: backend/algorithms/merge_rank.py

🔧 ALGORITHM:
  Sorts candidates by fusion_score using merge sort.
  Tracks rank changes: rank_delta = original_index - final_index.
  Positive delta = moved up in ranking, negative = moved down.
  
  Divide & Conquer:
    1. Split candidate list into halves recursively
    2. Merge two sorted halves while comparing fusion_scores
    3. Add rank_delta field to final sorted list

💡 VIVA ANSWER (2 sentences):
  "We used merge sort with rank delta tracking because we needed a stable O(n log n)
   sort that also captures how each candidate's ranking changed—useful for
   understanding ranking volatility in hiring decisions."

⏱️  COMPLEXITY:
  Time:  O(n log n) guaranteed (unlike quicksort)
  Space: O(n) for merge operations

📊 DEMO OUTPUT:
  Eve moved from rank 5 → 1 (Δ: +4)
  Alice moved from rank 1 → 4 (Δ: -3)
  (Higher fusion_scores = better final rankings)

🎓 QUICK RECALL FOR VIVA (1 minute answers)

Q: "Which algorithm did you use for candidate selection?"
A: "0/1 Knapsack DP. Company has a budget, each candidate has cost + score,
   we want max total score within budget. DP recurrence: dp[i][w] = max of
   (skip candidate i) or (include candidate i if budget allows)."

Q: "Why not use greedy for knapsack?"
A: "Greedy by score/cost ratio can fail on 0/1 Knapsack. Example: budget=10,
   candidate A (score=100, cost=9), candidate B (score=90, cost=5), C (score=85, cost=5).
   Greedy picks A (11.1 ratio), but optimal is B+C (total 175 vs A's 100).
   DP explores all subsets."

Q: "Why BFS and not DFS for skill graph?"
A: "Both explore the graph, but BFS finds shortest path (by level/distance).
   DFS might find a longer path first. Since we want minimum learning steps,
   BFS is better."

Q: "How does activity selection greedy work?"
A: "Sort by end time, then iterate. Pick each candidate whose start >= last_end.
   Why greedy is optimal: earliest-finishing activity leaves most room for
   future activities (exchange argument proof)."

Q: "Why merge sort over quicksort?"
A: "Guaranteed O(n log n) even worst-case (quicksort is O(n²) worst-case).
   Stable sort (preserves order of equal elements). Need extra space O(n)
   but worth it for guaranteed performance."
